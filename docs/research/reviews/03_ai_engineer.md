# Review: AI Engineer

**Reviewer:** ai-engineer specialist agent
**Date:** 2026-04-16
**Scope:** Full architecture + research folder, with focus on system quality, workflow correctness, citation fidelity, cascade dynamics, lint adequacy, evaluation, and the ingest→query→synthesize loop.

**Note on line references:** Line numbers cited in this review are approximate — the reviewer had read-only access via a line-aware viewer but did not consistently verify ranges. Treat numbers as regional rather than precise.

## 1. Executive Take

alexandria's architectural premise is defensible and, in several places, genuinely inspired. The decision to make compilation a write-time activity (cascade at ingest) rather than a query-time one is the correct bet for a single-user, local-first knowledge engine where the user's time is the dominant cost and the system can afford minutes-to-hours of background agent work. The three-layer separation (immutable raw sources, mutable LLM-owned wiki, deterministic schema) is cleanly argued, and the agent-as-retriever primitive set (`list`, `grep`, `search`, `read`, `follow`, `events`, `timeline`, `history`) maps directly onto what a frontier model already knows how to use from Claude Code. The research folder shows genuine engagement with the literature — the CatRAG critique (`docs/research/reference/13_agentic_retrieval_design_space.md:170-215`) is honest about where the no-vectors bet is load-bearing, and the mempalace comparison (`14_mempalace.md`) resists the temptation to dismiss a competitor. This is a design that has been thought about.

What concerns me most is that the system's correctness hinges on four LLM-in-the-loop mechanisms — cascade updates, citation fidelity, lint heuristics, and conversation synthesis — and at least three of those have no defense against the model getting it subtly wrong. The guardian agent is simultaneously the ingest worker, the cascade planner, the citation validator, the lint executor, and the synthesis engine, and the architecture has no adversarial check on its own output. There is also no evaluation scaffold at all — no benchmark, no regression suite, no "did the wiki get better or worse after this cascade" signal — which means the system will accumulate silent quality debt that no one can see until a retroactive query six months in returns something incoherent. The invariants are strong on intent but weak on falsifiability. That gap is the thing I would address before anything else.

## 2. The Loop Walkthrough: 50 Papers, Day 90, "What's the state of the art on X?"

Let me stress-test the actual flow.

**Ingest phase (days 1-89).** User runs `alexandria ingest paper_N.pdf` fifty times. Each invocation spawns the guardian, which reads the source, decides which `topics/` pages it affects, and applies cascade edits. Per `04_guardian_agent.md:188-260`, the ingest workflow is linear: read source → identify affected pages → plan edits → apply with exactly-one-match `str_replace` → validate → commit. The write ACL (doc 04:65-95) constrains writes to `wiki/**` only, citations are mandatory, and a pre-commit hook enforces the `[^N]: filename, p.X` footnote schema.

The first failure mode appears around paper 10-15: **cascade convergence.** Paper 10 contradicts paper 3. The guardian has to decide whether to rewrite the claim, add a "disputed" marker, or create a new sub-page. Nothing in doc 04 says what the right answer is. The write ACL lets the guardian do any of these. The lint checks (doc 04:385-460) flag contradictions but don't resolve them. Over 50 papers, you accumulate a page that either (a) silently converges on whichever view the last paper supported, or (b) becomes a dense thicket of hedges, or (c) fragments into a topic tree the user never asked for. None of these is wrong per se, but the architecture doesn't commit to one — which means the wiki's shape is a function of ingest order, not of the underlying knowledge. This is the thing that kills wikis, and the docs name it (01:95-110) without defending against it.

**Weekly synthesis (days 7, 14, 21, ...).** The temporal synthesis job (doc 10:120-180) produces `wiki/timeline/2026-W15.md` digests from event streams. Bounded budget, unattended. Per doc 10, these digests are themselves ingest events — they cascade into topic pages. So the weekly job is a second ingest path, running against a different kind of source (event streams) but landing in the same topic namespace. If the weekly synthesis is wrong — say, it over-indexes on a Slack thread that was noise — its errors also cascade. The docs don't describe a confidence or provenance differential between paper-derived claims and synthesis-derived claims. A future query can't tell if the line "the field is converging on approach X" came from a peer-reviewed paper or from a weekly digest of the user's own AI conversations.

**Day 90 query.** User asks Claude Code "what's the state of the art on X?" Claude Code, via MCP, hits alexandria primitives. The natural path is `search("X")` → `read(wiki/topics/x.md)` → `follow()` the footnotes to raw sources → `timeline()` for recent events. Invariant #13 (agent-as-retriever) says this should work because a frontier model knows how to compose these. I think that's right for the happy case. The problem is the unhappy case: **what if the topic page is stale, wrong, or internally inconsistent, and the agent reads it without knowing?** The agent sees a page with citations, trusts the citations, and returns an answer that looks well-grounded but is actually six weeks out of date because the last cascade missed an update. No primitive in the tool surface surfaces "this page is stale relative to its sources." No check says "the three citations on this claim actually disagree with the claim." The query works — but it works on a snapshot of the wiki's quality, and that quality is invisible.

The loop produces usable knowledge in the happy case and is undetectable when it degrades. That's the central weakness.

## 3. Specific Critiques

**3.1 Citation enforcement is syntactic, not semantic** (`docs/architecture/04_guardian_agent.md:267-310`). The guardrail is "every claim must have a footnote `[^N]: filename, p.X`" and the pre-commit hook parses the footnote format. The hook cannot check whether `paper.pdf, p.3` actually supports the claim. The guardian, being an LLM, can trivially hallucinate a plausible citation that doesn't map to the source text. The docs acknowledge the guardian is "an LLM, not an oracle" but offer no defense. Possible defenses — snippet-quoting (require the citation to include a verbatim quote), two-pass verification (a separate agent reads the cited span and votes), hash-locked quote anchors — are not discussed. Without one of these, the citation-mandatory rule is a compliance theater for the model to perform against itself.

**3.2 The exactly-one-match `str_replace` invariant is a correctness primitive, but it doesn't scale with cascade breadth** (`04_guardian_agent.md:144-170`). The primitive guarantees that edits are surgical, not accidental overwrites. Good. But a cascade that touches 12 pages requires 12 successful one-match replacements, each with its own planning step. If any one fails (because the page changed under the agent, or the target string wasn't unique), the cascade is half-applied. There is no transaction boundary. Doc 04:170 notes a commit happens at the end, but the file-system commit is not the same as atomicity of the cascade's semantic intent. A partial cascade is worse than no cascade — it silently violates the single-source-touches-one-page failure mode the invariants warn about.

**3.3 The lint split between deterministic and heuristic is drawn in the wrong place** (`04_guardian_agent.md:385-460`). Deterministic checks include citation format, header schema, broken links, orphaned files — all good. Heuristic checks include "claims without adequate provenance," "logical inconsistency," "stale sections." These are listed as "report only, agent decides what to fix." The problem is that an LLM running heuristic lint against its own prior writes has a strong confirmation bias — it wrote the page, and re-reading it, it will tend to justify what's there rather than flag it. The docs don't propose a hostile-reviewer posture (fresh context, adversarial prompt, no access to the write history). A lint agent that is literally the same model that wrote the text is not running lint, it's running review-at-peace-with-itself.

**3.4 `wiki_claim_provenance` is a structured mirror of the citation footnotes but is never used as a verification source** (`06_data_model.md:200-260`). Doc 06 describes the table as storing per-claim provenance with `source_file`, `source_page`, `claim_hash`, `created_at`. This is exactly the surface where semantic citation verification could live — you could store a verbatim quote alongside the hash and compare it to the source text on every re-read. The docs don't propose this. The table exists, but it's an index, not a check.

**3.5 The conversation capture design (doc 12) does not specify write conflict resolution for the hybrid SOURCE + EVENT_STREAM model** (`docs/architecture/12_conversation_capture.md`, general). A Claude Code session running in workspace A emits transcript events to alexandria. A second session in workspace B does the same. Both may, in the same minute, trigger synthesis that wants to update `wiki/topics/retrieval.md`. The guardian serializes writes (invariant: write ACL is single-writer), but there's no description of how the queue handles 30 concurrent sessions. Backpressure, prioritization, dedup — undefined. In practice the user will not have 30 sessions often, but 5-10 during a heavy work week is realistic, and doc 12 does not say what happens.

**3.6 The retroactive-query invariant (#15) is not testable** (`01_vision_and_principles.md`, invariant section). "Features are judged on whether they make knowledge six months from now more queryable" is a principle, not a metric. Nothing in the architecture surfaces a six-month test. You can't regression-test an invariant that can only be evaluated six months after the feature ships. Either the invariant needs a proxy (see §4) or it needs to be reframed as a design principle the team carries in its head, not a check the system enforces.

**3.7 The agent-as-retriever bet assumes tool-composition is the hard part, but for a wiki the hard part is topic decomposition** (`13_agentic_retrieval_design_space.md:170-215`, general framing). Karpathy's tweet and the Anthropic multi-agent post both describe retrieval over a stable corpus with known structure. alexandria's corpus is co-evolving with the queries — every ingest changes the index. A frontier model doing `search("X")` against a wiki that it itself wrote last week is doing something subtly different from an agent doing `search` against a fixed corpus. The research folder does not distinguish these cases and I think they are different.

**3.8 The cost model is not characterized** (general, all docs). Cascade at write time is cheaper over the lifetime of the wiki only if (queries × synthesis_cost_per_query) > (ingests × cascade_cost_per_ingest). Neither side is estimated. For a user who queries 5x/day and ingests 2x/day, with cascade touching on average 4 pages and synthesis touching 1-2, the crossover depends on token costs of each path. Doc 01:140-180 asserts the tradeoff without showing the math. This matters because if cascade costs are dominant (which they will be for small-topic wikis with high ingest rates), the architecture is losing money on every ingest and the user won't see it until the bill arrives.

## 4. Evaluation Proposal

Without these, the system is uncalibrated. I propose five metrics, ordered by importance:

**M1: Citation fidelity rate.** On a sample of 50 random claims drawn weekly from the compiled wiki, spawn a cold-context verifier agent (fresh session, no access to the guardian's write history) that reads the cited source and votes: supports / partially supports / does not support / source missing. Target: >95% supports on deterministic claims, >80% on synthesis-derived claims. This is the single most important metric — it directly tests the citation-as-compliance-theater failure mode.

**M2: Cascade coverage.** After each ingest, count the pages the guardian actually touched vs. the pages that SHOULD have been touched per a periodic ground-truth audit (e.g., grep the source's key terms across the wiki and check which pages mention them but were not updated). Target: >90% coverage. A declining trend is the early signal that cascade is being skipped under budget pressure.

**M3: Query resolution latency and success rate on a frozen benchmark.** Freeze 30 retroactive queries authored against the wiki at month 3, re-run them at months 4, 5, 6. Measure: (a) tool calls to resolution, (b) final-answer agreement with a human-labeled gold standard. This gives you a longitudinal signal on whether the wiki is getting more or less queryable over time — directly testing invariant #15.

**M4: Cost per ingest and per query, in tokens.** Track cumulative token spend on cascade vs. direct query. Compute the crossover week where cascade's amortized cost equals on-the-fly synthesis cost. This tells you whether the write-time compilation bet is winning.

**M5: Wiki self-consistency score.** Periodically sample pairs of claims on the same topic from different pages and ask a hostile verifier "do these agree?" Measure the contradiction rate. This is the proxy for the "wikis become graveyards" failure mode — the metric goes up when cascade is missing contradictions.

All five can be implemented as scheduled `alexandria lint` extensions — they fit the existing operation vocabulary — and all five produce signals that are actionable before six-month degradation is visible.

## 5. The Single Biggest Gap

**There is no adversarial check on the guardian's own output.** Every correctness mechanism — citation validation, cascade completeness, lint, synthesis — runs inside the same LLM context that did the write. The architecture assumes that because the guardian is a frontier model with good tool use, its self-review will be adequate. This is the assumption I am least willing to grant without evidence. The fix is not complicated: every write transaction should be verified by a second agent with (a) fresh context, (b) read-only access to the diff and the cited source spans, (c) a hostile prompt ("find the error in this write"). This costs roughly 2x inference per ingest, which the cost model currently does not budget for — but it is the difference between a system that degrades silently and one that catches its own failures before they compound.

This is also the gap that blocks every evaluation metric in §4 from being trustworthy. Without a hostile verifier, you cannot measure citation fidelity (the measurer is the same as the writer). Without a hostile verifier, cascade coverage audits are self-graded. The hostile-verifier pattern is the missing primitive.

## 6. Recommendations

**R1 (highest priority): Introduce a hostile verifier agent as a mandatory post-write step.** Fresh context, read-only, adversarial prompt. Writes are not committed until the verifier passes. This closes §3.1, §3.3, §3.4 in one stroke and makes every §4 metric meaningful.

**R2: Require verbatim-quote citations for every claim, stored in `wiki_claim_provenance` with a hash of the source span.** On re-read, the system can cheaply verify the quote still exists in the source. This is the semantic counterpart to the current syntactic citation check. It does not require LLM judgment to enforce.

**R3: Formalize cascade as a transaction with an explicit commit/rollback.** If the guardian's plan touches 12 pages and 11 succeed, roll back all 11. Half-cascades are worse than no cascade. This requires a staging area and an atomic commit — standard Git techniques, and doc 06 already uses SQLite so a transactional boundary exists in the wrong layer.

**R4: Build the evaluation scaffold (§4) before adding more capabilities.** Doc 10 (event streams) and doc 12 (conversation capture) both expand the ingest surface. Expanding ingest without evaluation is accelerating into an uncalibrated system. Freeze new ingest sources until M1 and M2 are wired up and producing weekly reports.

**R5: Decide explicitly on cascade convergence policy.** When paper N contradicts paper M, the guardian picks one of (rewrite, hedge, fragment). Pick the default and document it. My recommendation: hedge with a dated "as of paper N" marker, because it is the only option that preserves the retroactive-query property — a future query can ask "what did I think in April?" and get a real answer.

**R6: Characterize the cost model.** Estimate tokens per ingest (cascade-weighted), tokens per query, and the crossover point. Publish the assumptions in doc 01 or a new doc 13. This is the bare minimum to defend the "compile at write time" architectural choice on its own terms.

**R7: On the mempalace question — resist the hybrid temptation.** A vector layer added as "L3 recall fallback" is a commitment to maintaining two retrieval systems, two sources of truth, and two failure modes. The docs' current position (no vectors) is principled and testable. If §4's metrics show the agent-as-retriever bet failing after six months, revisit. Not before.

**R8: Minimum capability floor.** Run the ingest workflow against a weaker model (the docs mention Ollama as a hypothetical) on a fixed 10-source test set and measure §4's M1 and M2. Publish the floor. Users running below it should be warned at CLI startup. The architecture does not stop working below the floor — it fails silently, which is worse.

The design is sound in its bones. The principled bets are defensible. What the architecture currently lacks is a mechanism for knowing whether any of it is actually working, and a defense against the guardian being wrong in ways the guardian cannot see. Both gaps are closeable with modest additions that respect the existing invariants. I would close them before shipping another feature.
